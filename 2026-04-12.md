## 文章一：SQLite 3.53.0 发布

> 原文：[SQLite 3.53.0](https://simonwillison.net/2026/Apr/11/sqlite/)
> 来源：Simon Willison's Weblog | 分类：AI & ML

### 核心内容（中文翻译）

SQLite 3.53.0 是一个重要的版本更新，由于 3.52.0 版本被撤回，这个版本积累了大量的用户级和内部改进。以下是几个值得关注的亮点：

**1. ALTER TABLE 新增约束管理功能**

现在可以使用 `ALTER TABLE` 语句添加和删除 `NOT NULL` 和 `CHECK` 约束。这是一个非常实用的功能，此前需要使用 sqlite-utils 库的 `transform()` 方法来实现类似效果。

```sql
-- 添加 NOT NULL 约束
ALTER TABLE users ADD COLUMN email TEXT NOT NULL;

-- 移除约束
ALTER TABLE users DROP CONSTRAINT email_not_null;
```

**2. 新增 json_array_insert() 函数**

数据库现在原生支持 JSON 数组插入操作，同时提供 `jsonb` 等效版本，方便处理 JSON 数据格式。

**3. CLI 模式重大改进**

命令行工具增加了查询结果格式化功能。这个改进来自一个新的库——查询结果格式化器（Query Results Formatter）。Simon Willison 还使用 Claude Code 在手机上编译了 WebAssembly 版本，并构建了一个在线体验界面。

### 技术要点

- SQLite 虽然是轻量级数据库，但在嵌入式系统中应用广泛
- 约束管理功能简化了数据库迁移流程
- WebAssembly 编译使得桌面工具可以在浏览器中运行

### 对电赛的价值

在电赛项目中，如果需要使用本地数据存储（如参数配置、日志记录），SQLite 是一个不错的选择。相比于直接操作文件，SQLite 提供了更可靠的数据管理和查询能力。

## 文章二：WaitForMultipleObjects 句柄管理（续）

> 原文：[How do you add or remove a handle from an active Wait­For­Multiple­Objects?, part 2](https://devblogs.microsoft.com/oldnewthing/20260410-00/?p=112223)
> 来源：Microsoft DevBlogs - The Old New Thing | 分类：Systems

### 核心内容（中文翻译）

本文是 Raymond Chen 关于 Windows 多线程同步机制的深度技术文章。在上一篇文章中，我们讨论了如何异步添加或删除活动 `WaitForMultipleObjects` 的句柄，但异步操作可能导致一个问题：你可能删除了一个句柄并清理了该句柄依赖的资源，然后却收到该句柄已被触发的通知。

**解决方案：等待等待线程确认操作**

```cpp
_Guarded_by_(desiredMutex) DWORD desiredCounter = 1;
std::atomic<DWORD> activeCounter;

void wait_until_active(DWORD value)
{
    DWORD current = activeCounter;
    while (static_cast<int>(current - value) < 0) {
        activeCounter.wait(current);
        current = activeCounter;
    }
}
```

**工作流程**：

1. 后台线程修改 `desiredHandles` 和 `desiredActions`
2. 后台线程递增 `desiredCounter` 并触发事件
3. 等待线程被唤醒，处理变更
4. 等待线程发布当前变更计数器，让后台线程知道处理进度
5. 后台线程等待直到所有待处理的变更都被处理完毕

### 技术要点

- 使用原子操作避免竞态条件
- `WaitOnAddress` / `std::atomic::wait` 实现高效等待
- C++20 保证无符号整数到有符号整数转换的安全性
- 64位计数器可以避免溢出问题（32位计数器每1000次/秒需要约一个月才会溢出）

### 对电赛的价值

理解多线程同步机制对于 STM32 上的 FreeRTOS 开发非常重要：
- 任务间通信（队列、信号量、事件组）
- 资源保护（互斥量、临界区）
- 避免优先级反转问题

## 文章四：STM32 电机 PID 控制实战

> 来源：综合整理自多个开源项目和 CSDN 教程

### 核心内容（中文翻译）

**1. 硬件配置（典型方案）**

| 组件 | 型号 | 说明 |
|------|------|------|
| 主控芯片 | STM32F103C8T6 | ARM Cortex-M3, 72MHz |
| 电机驱动 | L298N | 双H桥驱动, 2A输出 |
| 编码器 | 霍尔编码器 | 13线/转, AB相输出 |
| 电机 | JGB37-520 | 12V直流减速电机 |

**2. 接线示意**

```
TIM2_CH1 ────────┬──> L298N ENA (PWM调速)
TIM2_CH2 ────────┤
                 │
PB12 ────────────┼──> L298N IN1 (方向控制)
PB13 ────────────┼──> L298N IN2
PB14 ────────────┼──> L298N IN3
PB15 ────────────┼──> L298N IN4
                 │
TIM3_CH1 ────────┴──> 编码器A相 (PA6)
TIM3_CH2 ───────────> 编码器B相 (PA7)

USART1_TX ──────────> USB-TTL RX (PA9)
```

**3. PID 参数配置**

```c
#define PID_KP  8.0f
#define PID_KI  0.5f
#define PID_KD  2.0f
#define CONTROL_PERIOD  50  // 控制周期（ms）
```

**4. 控制循环实现**

```c
void Control_Loop(void) {
    // 1. 读取当前转速
    current_rpm = Encoder_GetRPM(CONTROL_PERIOD);
    
    // 2. PID计算
    pid_output = PID_Compute_Position(&motor_pid, current_rpm);
    
    // 3. 输出到电机
    if (pid_output > 0) {
        Motor_SetSpeed(MOTOR_CW, (uint16_t)pid_output);
    } else if (pid_output < 0) {
        Motor_SetSpeed(MOTOR_CCW, (uint16_t)(-pid_output));
    } else {
        Motor.Stop();
    }
}
```

**5. HIL（硬件在环）仿真**

VirtuDrive-RTOS 项目展示了一个有趣的思路：使用 Python 模拟电机物理特性，STM32 控制"虚拟电机"而非真实电机。

```
STM32F446RE ──(PWM)──> Python仿真 ──(RPM反馈)──> STM32
```

这种方法的优点：
- 无需硬件即可调试 PID 参数
- 安全、可重复的测试环境
- 快速迭代算法

### 技术要点

- **编码器模式**：STM32 定时器支持编码器模式，自动计数
- **PWM 调速**：通过改变占空比控制电机转速
- **PID 整定**：可使用 Ziegler-Nichols 方法
- **防积分饱和**：设置积分项限幅

### 对电赛的价值

**直接相关**！电机控制是控制类电赛的核心：

1. **闭环控制**：使用编码器反馈实现精确速度控制
2. **参数可调**：通过串口或上位机动态调整 PID 参数
3. **实时性好**：DMA + 中断确保控制周期稳定

## 文章五：嵌入式图像处理优化

> 来源：综合整理自 arXiv 论文和技术博客

### 核心内容（中文翻译）

**1. 算法层面优化**

**简化复杂算法**：
- 减少迭代次数或递归调用
- 使用近似值替代精确计算
- 消除不必要的计算

例如：Canny 边缘检测计算复杂度高，可以用 Sobel 算子替代。

**查表法（LUT）和缓存**：
- 预计算昂贵的结果并存储
- 减少运行时计算开销

**2. 内存层面优化**

- **分块处理（Tile-based）**：将图像分成小块处理，减少内存带宽需求
- **流式处理**：最小化内存占用
- **ROI 处理**：只处理感兴趣区域

```c
// Q15 定点化示例（比浮点快2.3倍）
// 浮点版本
float kernel[9] = {0.0625f, 0.125f, 0.0625f, ...};

// Q15定点版本
int16_t kernel_q15[9] = {2048, 4096, 2048, ...};
```

**3. 代码层面优化**

- **循环展开**：减少循环控制开销
- **数据对齐**：改善内存访问模式
- **SIMD 指令**：利用 NEON/DSP 并行处理

**实测效果**（Sobel 边缘检测）：
- 1920×1080 图像处理：42ms → 7.3ms
- DDR 带宽利用率：41% → 89%

### 对电赛的价值

如果电赛涉及视觉处理（如巡线、目标识别）：

1. **定点运算**：STM32F4 建议使用 Q15 定点化
2. **分级精度**：平衡速度和精度
3. **DMA 传输**：摄像头数据用 DCMI + DMA
