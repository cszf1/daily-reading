## 文章一：C++ 编译器错误解读：代码里明明没有 "->" 为什么报错

> 原文：[Learning to read C++ compiler errors: Illegal use of -> when there is no -> in sight](https://devblogs.microsoft.com/oldnewthing/20260406-00/?p=112208)
> 来源：devblogs.microsoft.com/oldnewthing | 分类：Systems

### 核心内容

一位客户报告了系统头文件的问题。当他们包含 `ole2.h` 时，编译器在 `oaidl.h` 中报错：

```
oaidl.h(5457,43): error C3927: '->': trailing return type is not allowed after a non-function declarator
oaidl.h(5457,43): error C3613: missing return type after '->' ('int' assumed)
oaidl.h(5457,43): error C3646: 'Log': unknown override specifier
oaidl.h(5457,43): error C2275: 'LPCOLESTR': expected an expression instead of a type
oaidl.h(5457,43): error C2146: syntax error: missing ')' before identifier 'pszPropName'
oaidl.h(5459,60): error C2238: unexpected token(s) preceding ';'
```

**关键发现**：编译器在报"幽灵错误"——它抱怨的是代码中不存在的东西，比如 `->` 和 `Log`。

### 技术要点

1. **当编译器报错说代码里有不存在的符号时，首先要怀疑是宏在作祟**。宏可以在编译前向代码中插入任意字符。

2. **解决方法**：让编译器生成预处理后的文件，这样可以看清宏展开后的真实代码。

3. **教训**：编译器虽然有时报错信息晦涩，但它从不会无中生有。如果它说 `->` 被误用了，那一定是真的有 `->` 被误用了（通常是某个宏展开后产生的）。

### 对电赛的价值

在嵌入式开发中（STM32等），我们经常面对各种编译器报错。这篇文章的启示是：
- 不要只看报错信息表面的位置，**思考是否有宏在暗中捣乱**
- 学会使用预处理输出（gcc -E）来诊断问题
- 养成查看头文件中宏定义的好习惯

## 文章二：AI 12分钟写完代码，我花了10小时修复

> 原文：[AI Did It in 12 Minutes. It Took Me 10 Hours to Fix It](https://idiallo.com/blog/it-took-me-10-hours-to-fix-ai-code?src=feed)
> 来源：idiallo.com | 分类：Indie | 点赞：37

### 核心内容

作者用 AI（LMM）生成了一个 PHP 媒体管理器应用：
- AI 在 **12分钟** 内生成了约 5000 行代码
- 但代码是"意面代码"（spaghetti code）——HTML、CSS、数据库查询全塞在一个文件里
- 作者花了 **10小时** 来修复和重构

### AI 生成代码的问题

1. **代码质量参差不齐**：生成的内容往往过于冗长，很多代码块根本用不到
2. **难以理解和维护**：如果不能理解代码的工作原理，当它出问题时就无法修复
3. **系统提示词冲突**：AI 的系统提示可能与你的具体需求矛盾

### 重构后的项目结构

```
media-manager/
├── config.php
├── init.php
├── public/
│   └── index.php
├── src/
│   ├── controllers/    # 控制器
│   ├── lib/           # 工具库
│   └── models/        # 数据模型
└── template/
    └── layout.php
```

重构后：**22个文件，1254行代码** ——比原来简洁得多。

### 关键感悟

> "如果我没有应用如何工作的心智模型，当它失败时我能做什么？"

- AI 给了"快速完成"的错觉
- **清理和生成一样重要**，否则代码最终会失控
- 对于需要长期维护的项目，**理解代码比生成代码更重要**

### 对电赛的价值

电赛备赛时间紧张，AI 辅助编程很诱人，但要记住：
- **AI 生成的代码必须经过自己的理解和验证**
- 不要为了"用 AI"而放弃对底层原理的学习
- 对于嵌入式系统，**硬件和控制逻辑的确定性比快速实现更重要**
- 建议：用 AI 生成框架，但核心控制算法自己写
