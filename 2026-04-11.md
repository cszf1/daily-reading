## 文章一：C++编译器错误诊断：当你看到不存在的箭头符号时

> 原文：[Learning to read C++ compiler errors: Illegal use of -> when there is no -> in sight](https://devblogs.microsoft.com/oldnewthing/20260406-00/?p=112208)
> 来源：Microsoft DevBlogs | 分类：Systems

### 核心内容

**问题现象**

一位客户报告了一个系统头文件的问题。当他们包含 `ole2.h` 时，编译器在 `oaidl.h` 中报告了错误。编译器报错说代码中有非法的 `->` 使用，但实际上代码中根本没有这个符号——编译器在"看到鬼"。

原始代码：
```cpp
MIDL_INTERFACE("3127CA40-446E-11CE-8135-00AA004BB851")
IErrorLog : public IUnknown
{
public:
 virtual HRESULT STDMETHODCALLTYPE AddError( // error here
 /* [in] */ __RPC__in LPCOLESTR pszPropName,
 /* [in] */ __RPC__in EXCEPINFO *pExcepInfo) = 0;
};
```

编译器报告的错误包括：
```
oaidl.h(5457,43): error C3927: '->': trailing return type is not allowed after a non-function declarator
oaidl.h(5457,43): error C3613: missing return type after '->' ('int' assumed)
oaidl.h(5457,43): error C3646: 'Log': unknown override specifier
oaidl.h(5457,43): error C2275: 'LPCOLESTR': expected an expression instead of a type
oaidl.h(5457,43): error C2146: syntax error: missing ')' before identifier 'pszPropName'
```

**根本原因**

作者 Raymond Chen 指出：**当你看到编译器报告代码中不存在的东西时，应该怀疑是宏**。因为宏可以在代码中插入字符。

在这个案例中，编译器怀疑有一个名为 `AddError` 的宏，其展开结果包含了 `->` 符号。但客户声称自己没有任何这样的宏。

**解决方案**

作者建议生成预处理文件，查看编译器实际处理的内容。结果客户发现——原来确实有一个 `AddError` 宏！禁用该宏后问题解决。

**经验总结**

> "编译器对于错误信息的描述有时会很晦涩，但据我所知，它并不是恶意的。如果它抱怨你错误地使用了 `->`，那么很可能是真的有一个 `->` 被错误地使用了。"

### 技术要点

- **宏展开问题**：C/C++ 预处理器在编译前进行文本替换，可能导致编译器报错位置与实际问题位置不一致
- **预处理检查**：使用编译器生成的预处理文件（.i）来追踪宏展开后的实际代码
- **调试技巧**：当你怀疑有宏干扰时，可以定义宏为空或使用 `#undef` 来排除干扰

### 对电赛的价值

作为电赛参赛者，你正在学习 C++（特别是类与对象部分）。理解编译器错误信息非常重要：
1. **预处理阶段不可忽视**：宏在STM32开发中很常用（如 `HAL_GPIO_WritePin` 等）
2. **学会阅读编译器输出**：编译错误往往指向的是症状而非根本原因
3. **宏命名冲突**：实际项目中可能遇到第三方库与你自己的宏冲突的情况

## 文章二：用树莓派搭建你自己的拨号ISP

> 原文：[Build your own Dial-up ISP with a Raspberry Pi](https://www.jeffgeerling.com/blog/2026/build-your-own-dial-up-isp-with-a-raspberry-pi/)
> 来源：Jeff Geerling Blog | 分类：Hardware

### 核心内容

**项目概述**

作者 Jeff Geerling 用树莓派搭建了一个复古的拨号ISP，实现了用1999年的 Apple iBook G3 通过 Wi-Fi 连接到古老的拨号网络。

**硬件组成**

| 组件 | 价格 | 说明 |
|------|------|------|
| SBC (树莓派 3/4/5) | $40 | 运行 Linux 的单板计算机 |
| 电话线路模拟器 (Viking DLE-200B) | $120 | 模拟两台调制解调器之间的通话 |
| USB 56K 调制解调器 | $45 | StarTech.com USB 调制解调器 |

**软件架构**

系统使用两个核心 Linux 工具：

1. **mgetty** ("modem get tty")
   - 处理调制解调器的来电
   - 与远程调制解调器协商连接
   - 连接建立后交给 PPP

2. **PPP** (Point-to-Point Protocol)
   - 对远程计算机进行身份验证
   - 配置网络桥接
   - 让远程计算机表现得像在本地网络中

**连接日志示例**：
```
Mar 26 15:32:35 dialpi pppd[15926]: PAM Account OK for dial
Mar 26 15:32:35 dialpi pppd[15926]: user dial logged in on tty ttyACM0 intf ppp0
Mar 26 15:32:35 dialpi pppd[15926]: PAP peer authentication succeeded for dial
Mar 26 15:32:35 dialpi pppd[15926]: Peer dial authenticated with PAP
```

**技术细节**

- 调制解调器使用 **QAM（正交振幅调制）** 技术来传输数据
- 现代 WiFi 正是基于相同的 QAM 原理，只是扩展到了更高的带宽
- 最稳定的连接速度约为 28.8Kbps，偶尔能达到 33.6Kbps

**Wi-Fi 桥接**

最有趣的部分是将复古的 AirPort Wi-Fi（802.11b，最大11Mbps）与拨号网络结合起来，让1999年的 iBook 可以"无线拨号上网"。

### 技术要点

- **mgetty + PPP**：经典的 Linux 拨号服务器方案
- **PAP 认证**：Password Authentication Protocol，PPP 协议的一部分
- **QAM 调制**：从电话调制解调器到现代 WiFi 的技术演进
- **串口通信**：使用 AT 命令集与调制解调器交互

### 对电赛的价值

这个项目展示了嵌入式系统的 **通信协议实战**：
1. **理解协议栈**：PPP 协议展示了点对点通信的实现方式
2. **硬件接口**：USB 调制解调器通过串口与树莓派通信，这是 UART 通信的典型应用
3. **网络配置**：展示了如何配置网络桥接和身份验证
4. **调试技能**：使用 `minicom` 直接与调制解调器交互，使用 `journalctl` 查看系统日志

这些技能在电赛的控制系统调试中同样重要——理解通信协议、串口调试、日志分析。

## 文章三：watgo - Go语言的 WebAssembly 工具包

> 原文：[watgo - a WebAssembly Toolkit for Go](https://eli.thegreenplace.net/2026/watgo-a-webassembly-toolkit-for-go/)
> 来源：Eli Bendersky's Blog | 分类：Systems

### 核心内容

**项目简介**

作者发布了 **watgo** - 一个纯 Go 语言实现的 WebAssembly 工具包，类似于 wabt (C++) 或 wasm-tools (Rust)，但不依赖任何外部库。

**核心功能**

```
┌─────────────┐    Parse    ┌─────────────┐    Validate    ┌─────────────┐
│  WAT 文本格式 │ ────────→  │   wasmir    │ ───────────→  │   wasmir    │
│             │             │ (语义表示)   │               │ (已验证)    │
└─────────────┘             └─────────────┘               └─────────────┘
         ↑ Encode                          Decode                    │
         │                      ┌─────────────┐                     │
         └──────────────────────│   WASM 二进制 │ ←─────────────────┘
                                └─────────────┘
```

**wasmir 语义表示**

`wasmir` 是 WebAssembly 模块的语义表示，提供易于使用的 API：

```go
package main

import (
  "fmt"
  "github.com/eliben/watgo"
  "github.com/eliben/watgo/wasmir"
)

const wasmText = `
(module
 (func (export "add") (param i32 i32) (result i32)
  local.get 0
  local.get 1
  i32.add
 )
)`

func main() {
  m, err := watgo.ParseWAT([]byte(wasmText))
  if err != nil {
    panic(err)
  }

  i32Params := 0
  localGets := 0
  i32Adds := 0

  for _, fn := range m.Funcs {
    sig := m.Types[fn.TypeIdx]
    for _, param := range sig.Params {
      if param.Kind == wasmir.ValueKindI32 {
        i32Params++
      }
    }

    for _, instr := range fn.Body {
      switch instr.Kind {
      case wasmir.InstrLocalGet:
        localGets++
      case wasmir.InstrI32Add:
        i32Adds++
      }
    }
  }

  fmt.Printf("module-defined funcs: %d\n", len(m.Funcs))
  fmt.Printf("i32 params: %d\n", i32Params)
  fmt.Printf("local.get instructions: %d\n", localGets)
  fmt.Printf("i32.add instructions: %d\n", i32Adds)
}
```

**测试策略**

watgo 使用了 WebAssembly 官方的完整测试套件（近200K行 WAT 文件）进行端到端测试，并通过了整个 WASM spec 核心测试套件。

### 技术要点

- **零依赖设计**：纯 Go 实现，无外部依赖
- **WAT → WASM**：WebAssembly 文本格式到二进制格式的转换
- **语义验证**：遵循官方 WebAssembly 验证语义
- **编译器原理**：展示了词法分析、语义分析、代码生成的完整流程

### 对电赛的价值

虽然 WebAssembly 主要用于 Web 开发，但这个项目展示了 **编译器工具链的开发思路**：

1. **词法/语法分析**：理解程序的文本表示和结构化表示之间的转换
2. **验证逻辑**：对输入进行合法性检查，确保安全
3. **模块化设计**：Parse → Validate → Encode → Decode 的清晰架构
4. **测试驱动**：使用官方测试套件保证正确性

对于嵌入式开发，理解编译器后端和工具链有助于你在 STM32 开发中更好地理解编译过程、调试信息格式等。
