## 文章一：C++编译器错误解读：代码里明明没有「->」却报错

> 原文：[Learning to read C++ compiler errors: Illegal use of -> when there is no -> in sight](https://devblogs.microsoft.com/oldnewthing/20260406-00/?p=112208)
> 来源：Microsoft DevBlogs - The Old New Thing | 分类：Systems

### 核心内容

这是一篇关于如何阅读 C++ 编译器错误信息的文章，作者是 Raymond Chen（Windows 开发领域超过 30 年经验）。

**问题描述**

一位客户报告了一个系统头文件问题：当他们 `#include <ole2.h>` 时，编译器在 `oaidl.h` 中报错：

```
oaidl.h(5457,43): error C3927: '->': trailing return type is not allowed after a non-function declarator
oaidl.h(5457,43): error C3613: missing return type after '->' ('int' assumed)
oaidl.h(5457,43): error C3646: 'Log': unknown override specifier
oaidl.h(5457,43): error C2275: 'LPCOLESTR': expected an expression instead of a type
oaidl.h(5457,43): error C2146: syntax error: missing ')' before identifier 'pszPropName'
oaidl.h(5459,60): error C2238: unexpected token(s) preceding ';'
```

**关键洞察：编译器在"看到不存在的东西"**

编译器报错说代码中有 `->` 和 `Log`，但查看源代码后发现这些内容根本不存在！

**原因分析：宏在作怪**

当编译器报告代码中存在"不存在的内容"时，应该怀疑是宏。宏可以在预处理阶段向代码中插入字符。

作者怀疑存在一个名为 `AddError` 的宏，其展开结果包含了 `->` 符号。

客户起初声称没有这个宏，但当作者要求生成预处理文件（preprocessor output）后，客户发现确实存在一个 `AddError` 宏。禁用该宏后问题解决。

### 技术要点

- **预处理文件是关键**：当编译器报错与源代码不匹配时，生成预处理文件（.i）查看宏展开后的实际代码
- **编译器通常是对的**：如果编译器抱怨 `->` 被误用，那么很可能是真的被误用了——只是通过宏的方式
- **调试策略**：编译器报错可能是间接的，真正的问题可能藏在宏定义里

### 对电赛的价值

这篇文章虽然讲的是 Windows 开发场景，但传达了一个通用原则：**当代码行为与预期不符时，不要只看表面代码，要检查预处理/编译过程**。对于嵌入式开发来说，这点尤其重要——你可能会遇到各种宏定义、条件编译导致的奇怪问题。

## 文章二：IBM航天计算机的兴衰：图解历史

> 原文：[The rise and fall of IBM's 4 Pi aerospace computers: an illustrated history](http://www.righto.com/2026/03/ibm-4-pi-computer-history.html)
> 来源：Ken Shirriff's Blog | 分类：Hardware

### 核心内容

这篇文章详细介绍了 IBM System/4 Pi 系列计算机的历史，这是专为航空电子设备设计的紧凑型高性能计算机系列。

**背景**

1964 年 IBM 推出 System/360 系列大型机，彻底改变了计算机行业。1967 年左右，IBM 推出了 System/4 Pi 系列，将这一理念扩展到三维应用领域——4π 代表一个完整球面的立体角数量。

**第一代产品**

1. **TC战术计算机（TC Tactical Computer）**
   - 设计用于卫星、战术导弹、直升机等需要小型轻量计算机的场景
   - 16位或32位字长，8KB到64KB磁芯内存
   - 包含54条指令（含乘除法）
   - 最初性能：48,500指令/秒

2. **CP定制处理器（CP Customized Processor）**
   - 16位CPU，36位宽数据总线
   - 可选的微码内部编程，指令集可定制
   - 性能：91,000指令/秒

3. **EP扩展性能计算机（EP Extended Performance）**
   - 32位计算机，兼容 IBM System/360
   - 微码编程，支持复杂的360指令集
   - 性能：190,000指令/秒

** Skylab 的数字控制**

TC-1 计算机在 1973 年发射的美国第一个空间站 Skylab 中发挥了关键作用。两个 TC-1 计算机控制三个巨型陀螺仪（每个155磅）来精确控制空间站姿态，避免消耗推进剂。**这是载人航天器上第一个全数字控制系统。**

**航天飞机计算机**

约 1975 年，IBM 为航天飞机开发了 AP-101B 计算机。航天飞机配备了**四台冗余 AP-101B 计算机**，并行运行并对每个输出进行投票表决，第五台计算机作为备份。这种冗余设计确保了极高的可靠性。

**技术演进**

| 型号 | 年份 | 性能 (MIPS) | 特点 |
|------|------|-------------|------|
| TC-1 | ~1970 | 0.06 | 磁芯内存，首个全数字控制 |
| AP-101B | ~1975 | 0.45 | 航天飞机主力计算机 |
| AP-101S | ~1985 | 1.27 | 三倍性能提升，重量减半 |
| AP-102 | ~1985 | 1+ | VHSIC芯片，12,000门/VLSI |

**关键工程经验**

1. **冗余设计**：四冗余投票机制确保单点故障不影响任务
2. **容错能力**：核环境检测与快速重启（50毫秒恢复）
3. **软硬件协同**：从汇编到高级语言（JOVIAL）的演进
4. **摩尔定律的挑战**：军事计算机从立项到服役可能需要十年，期间商业计算机性能可能提升一个数量级

### 技术要点

- **磁芯内存**：非易失性、抗辐射，是太空应用的理想选择
- **模块化设计**：使用"page"（页）结构的电路板，便于维护和升级
- **MIL-STD-1750A**：1980年美国空军发布的16位标准架构，强制要求未来项目采用
- **VHSIC计划**：Very High Speed Integrated Circuit，美国军方亿元级投资用于保持对苏联的技术优势

### 对电赛的价值

这篇文章虽然讲的是航空航天计算机，但揭示了控制系统的核心工程原则：

1. **实时性**：航天飞机计算机需要在毫秒级响应
2. **可靠性**：冗余设计、容错机制是控制系统的命脉
3. **迭代演进**：从 TC-1 到 AP-101S 的演进路径，对于理解控制系统从原型到产品化很有参考价值

对于准备控制类电赛的同学，理解这些航空航天控制系统的设计思想，有助于设计出更可靠、更稳定的控制系统方案。

## 文章三：从零构建LLM（32i）：噪声中藏着什么

> 原文：[Writing an LLM from scratch, part 32i -- Interventions: what is in the noise?](https://www.gilesthomas.com/2026/04/llm-from-scratch-32i-interventions-what-is-in-the-noise)
> 来源：Giles Thomas | 分类：Indie

### 核心内容

这是作者从零构建 GPT-2 规模 LLM 系列文章的的第32篇，探讨训练过程中的随机性对最终结果的影响。

**实验设置**

作者在 8xA100 40GB GPU 上训练一个 163M 参数的 GPT-2 风格模型，使用完全相同的代码和数据，进行多次训练运行，对比不同随机种子（seed）下的结果。

**惊人发现：完全确定性的训练**

两次完全相同的训练运行（相同代码、相同数据、相同机器，只是时间不同），产生了**比特级完全相同**的模型权重！

```bash
# 两次训练的测试集损失
Run 1: 3.691526
Run 2: 3.691526
```

这说明 PyTorch + CUDA 的随机数生成在分布式训练中是**完全确定性**的。

**随机种子实验**

| 随机种子 | 测试集损失 | 与基线相比 |
|----------|------------|------------|
| 42 (基线) | 3.691526 | - |
| 22 | 3.673453 | +0.018 改进 |
| 67 | 3.653593 | +0.038 改进 |

**关键发现：权重初始化是噪声的主要来源**

作者进行了更细致的实验，将随机性分为两部分：
1. **模型权重初始化**：设置 seed X
2. **训练过程**：设置 seed Y

| 实验类型 | 均值 | 标准差 |
|----------|------|--------|
| 固定权重初始化，仅改变训练随机性 | 3.684462 | 0.008672 |
| 固定训练随机性，仅改变权重初始化 | 3.673215 | 0.012062 |

**结论**：训练过程中的随机性（dropout 等）影响很小，而**模型权重的初始化**才是导致结果差异的主要原因。

### 技术要点

- **确定性训练**：现代深度学习框架（PyTorch + CUDA）在特定配置下可以实现完全确定性训练
- **Dropout 的影响被高估**：即使 dropout 在训练中丢弃 10% 的激活，其对最终结果的影响远小于权重初始化的影响
- **权重初始化策略**：选择好的权重初始化方法可能比调整训练超参数更重要
- **分布式训练的确定性**：使用 `torch.manual_seed()` 和 `torch.cuda.manual_seed_all()` 可以实现跨 GPU 的确定性

### 对电赛的价值

这篇文章虽然讲的是 AI 模型训练，但涉及的**随机性控制和确定性行为**对嵌入式和控制系统开发有重要启示：

1. **可复现性**：在调试控制系统时，确保每次运行的初始条件一致非常重要
2. **权重初始化的重要性**：类似于控制系统中 PID 参数的初始值对最终控制效果的影响
3. **边缘计算趋势**：随着 AI 芯片（如 NVIDIA Jetson、Google Edge TPU）的发展，在嵌入式设备上运行 AI 模型成为可能，这篇文章的实验方法论值得参考
